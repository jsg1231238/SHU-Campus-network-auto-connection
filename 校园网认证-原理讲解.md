## 1. 网络分层：连上 WiFi 却不能上网，卡在哪一层

网络通信是**分层**的。粗略地说：

- **第二层（数据链路层）**：设备之间用 **MAC 地址**（网卡的 48 位硬件编号，如 `9c:b1:50:12:1c:81`）互相识别，管的是"同一个局域网内的帧传输"。连上 WiFi = 完成了第二层的关联（association）。
- **第三层（网络层）**：用 **IP 地址**寻址、跨网段路由。你连上 WiFi 后拿到一个 IP（本机 `203.0.113.45`），具备了第三层能力。
- **第七层（应用层）**：HTTP 等协议，浏览器和网站在这层对话。

**门户认证（Captive Portal）的本质**：你在第二、三层都通了（有 MAC、有 IP），但中间的**网关设备**（锐捷这里是无线接入控制器 AC / 网络接入服务器 NAS，型号 `YC_RG-N18012`）维护着一张"哪些 MAC/IP 已认证"的表。未认证时，它在第三 / 七层做**策略拦截**：

- 只放行到认证服务器 `10.10.9.9` 的流量；
- 对其它目标的 HTTP 请求，**劫持并重定向**到认证页；
- 你认证成功后，它把你这条会话（由 `wlanuserip` + `mac` + 接入端口 `nasportid` 等唯一确定）标记为放行。

所以脚本要做的，就是**在应用层伪装成浏览器，完成一次登录请求**，让 AC 改写那张表。理解这点就明白：为什么 `wlanuserip`、`mac`、`nasid`、`nasportid` 这些参数是认证的核心——它们精确描述"你是哪条接入会话"。

## 2. HTTP 协议：请求与响应的确切结构

HTTP 跑在 TCP 之上（TCP 负责可靠字节流，HTTP 定义字节流里的"文本报文"格式）。一次请求报文的实际字节长这样（`\r\n` 是回车换行，是协议规定的分隔符）：

```http
POST /eportal/InterFace.do?method=login HTTP/1.1\r\n
Host: 10.10.9.9\r\n
Content-Type: application/x-www-form-urlencoded; charset=UTF-8\r\n
Cookie: JSESSIONID=965AE3F8...\r\n
Referer: http://10.10.9.9/eportal/index.jsp?...\r\n
\r\n
userId=XXXXXXXX&password=<RSA密文十六进制>&service=shu&...
```

结构三段：

1. **请求行**：方法 + 路径（含查询字符串）+ 协议版本。
2. **请求头（headers）**：若干 `名: 值`，一行一个。描述元信息。
3. **空行 + 请求体（body）**：POST 才有；上面的表单数据就是 body。

响应报文对称：状态行（`HTTP/1.1 200 OK`）+ 响应头 + 空行 + 响应体。

### 方法语义

- **GET**：获取资源，无 body，参数在 URL 里。语义上"安全、幂等"（不改服务器状态）。
- **POST**：提交数据，有 body，语义上"非幂等"（会改状态，如登录、下单）。

### 状态码（本项目遇到的）

- `200 OK`：成功。**注意**：网关劫持时也用 200 返回一段"跳转 JS"，而非标准的 302——这是个坑（见第 3 节）。
- `302 Found` / `301 / 307`：重定向，`Location` 头给出新地址。`http://10.10.9.9/` 直接访问就回 302 → `redirectortosuccess.jsp`。

### `Content-Type` 的关键作用

它声明 body 的媒体类型与字符集。我们登录用 `application/x-www-form-urlencoded`——即"表单"格式：`键=值` 用 `&` 连接、值做百分号编码。**服务器按这个头决定怎么解析 body**。反过来，响应**没有** `Content-Type` 头时，客户端不知道该按文本还是二进制处理——这正是阶段 5 那个 byte[] 坑的根源（第 7 节详述）。

## 3. 重定向劫持：为什么探测裸 IP 能拿到认证参数

未认证时，脚本主动 `GET http://123.123.123.123/`（一个随便的公网 IP）。AC 拦下这个请求，回给我们的**不是** 302，而是 `200` + 这样一段响应体：

```html
<script>top.self.location.href='http://10.10.9.9/eportal/index.jsp?wlanuserip=203.0.113.45&wlanacname=YC_RG-N18012-Co&ssid=&nasip=172.18.2.117&mac=aabbccddeeff&...&nasportid=TenGigabitEthernet 2/4/28.04020000:402-0'</script>
```

这段 JS 让**浏览器**跳转到认证页。关键：URL 里的这些参数是 **AC 在劫持这一刻、根据你当前接入会话动态生成并注入的**：

- `wlanuserip`：AC 给你分配的 IP；
- `mac`：你网卡的 MAC；
- `nasip` / `nasid` / `wlanacname`：接入设备标识；
- `nasportid`：你接入的物理端口（`TenGigabitEthernet 2/4/28...`）。

**为什么必须实时抓、不能写死**：这些是会话级的，重连 / 换设备 / 换端口都会变。而且——`mac` 之后还会被拼进加密密码里（第 6 节），所以它必须来自"本次真实会话"。

脚本 `Get-RedirectQueryString` 做的就是：主动触发一次劫持 → 从响应体里用正则抠出 `index.jsp?` 后面那一整串（即最新 queryString）。

## 4. 百分号编码与"双重编码"：RFC 3986 与嵌套的陷阱

### 4.1 为什么要编码

URL 里有一组**保留字符（reserved）**有语法含义：`: / ? # [ ] @ ! $ & ' ( ) * + , ; =`。其中 `&`、`=` 是查询字符串的分隔符。若某个"值"里本身含这些字符，或含空格 / 非 ASCII，就会破坏结构。

**百分号编码（percent-encoding，RFC 3986）**：把一个字节写成 `%` + 两位十六进制。空格 → `%20`，`=` → `%3D`，`&` → `%26`，`/` → `%2F`，`%` 自身 → `%25`。**未保留字符（unreserved）** `A-Z a-z 0-9 - _ . ~` 不编码。

> 细节：JS 的 `encodeURIComponent` 与 .NET 的 `[uri]::EscapeDataString` 对未保留集略有差异（前者不编码 `!'()*`，后者会编码它们），但本项目的值只含十六进制、字母数字和上述保留字符，两者结果一致，可互换。

### 4.2 为什么"编码两次"

登录请求里，**整个 queryString（内部已有 `&`、`=`）要作为外层表单的一个值** `queryString=...` 塞进去。若只编码一次，内层的 `&`、`=` 仍会被外层解析器当成分隔符，结构就乱了。于是要**再编码一层**，把第一层产生的 `%` 也转义掉：

$$
\text{mac=aabbccddeeff}
\xrightarrow{\text{编码1}} \text{mac\%3Daabbccddeeff}
\xrightarrow{\text{编码2}} \text{mac\%253Daabbccddeeff}
$$

浏览器就是 `encodeURIComponent(encodeURIComponent(qs))`，脚本对应 `[uri]::EscapeDataString([uri]::EscapeDataString($qs))`。**外层解析器解一次 → 还原成"含 %3D 的字符串"（仍是一个整体值）→ 服务器再解一次 → 拿到原始 queryString。** 两端约定层数必须一致。

### 4.3 空格截断 bug 的深层原因（阶段 7）

AC 那段 JS 里 `nasportid=TenGigabitEthernet 2/4/28...` 含一个**真实空格字节（0x20）**。浏览器把这段 URL 载入地址栏时，会按 URL 规范**把空格规范化成 `%20`**，于是 `document.location.search` 里是 `%20`，再双重编码得到 `%2525…`。

而我们是**直接从响应文本抠字符串**，拿到的是那个真实空格。旧正则 `[^…\s…]+`（排除空白）遇到空格就停 → queryString 被截断。修法两步：正则改为"排除引号 / 尖括号"到 URL 结束才停 `([^'"<>\r\n]+)`；捕获后 `-replace ' ', '%20'` 手动补上，**复现浏览器的规范化**。这样双重编码后与浏览器逐字节一致。

## 5. Cookie 与会话：无状态协议如何"记住你"

HTTP 本身**无状态（stateless）**：每个请求独立，服务器默认不知道两个请求来自同一个人。**Cookie** 是补丁：

- 服务器在响应里用 `Set-Cookie: JSESSIONID=abc; Path=/eportal` 下发；
- 浏览器存下，之后对该域 / 路径的每个请求自动带 `Cookie: JSESSIONID=abc`。

`JSESSIONID` 是 Java Servlet 容器（如 Tomcat）的**会话标识**：服务器端为每个会话开一块内存，用这个 ID 索引，把"跨请求的中间状态"存在那里。抓包里出现**两个 `JSESSIONID`**，通常是不同路径（`/` 与 `/eportal`）各一个。

本项目里，脚本先 `GET index.jsp` 拿一枚新的 `JSESSIONID`，再带着它 POST 登录，模拟浏览器"先打开页面、后提交"的会话连续性。`EPORTAL_COOKIE_USERNAME/PASSWORD/SERVER` 则是**浏览器端 JS 写的**"记住密码"Cookie，非服务器会话所必需（首次登录本就没有它们）。

## 6. RSA 非对称加密：数学、这套实现的非标准之处、以及为何能"离线证明"

### 6.1 对称 vs 非对称

- **对称加密**：加解密同一把密钥，快，但密钥分发是难题。
- **非对称加密**：一对密钥。**公钥**加密、**私钥**解密，二者由数学难题关联，从公钥无法（在可行时间内）反推私钥。RSA 是最经典的一种。

### 6.2 RSA 数学

密钥生成：取两个大质数 \( p, q \)；令 \( n = pq \)，欧拉函数 \( \varphi(n) = (p-1)(q-1) \)；选公开指数 \( e \) 使 \( \gcd(e, \varphi(n)) = 1 \)；求私钥 \( d \equiv e^{-1} \pmod{\varphi(n)} \)。

- **公钥** = \( (e, n) \)，**私钥** = \( d \)。
- 加密：明文数字 \( m \)（需 \( m < n \)），密文
$$ c \equiv m^{e} \pmod{n}. $$
- 解密：
$$ m \equiv c^{d} \pmod{n}. $$

正确性来自欧拉定理：因 \( ed \equiv 1 \pmod{\varphi(n)} \)，故 \( m^{ed} \equiv m \pmod{n} \)。**安全性**在于：已知 \( n \) 要还原 \( d \)，等价于分解 \( n = pq \)；当 \( n \) 为 1024 位时大数分解在计算上不可行。

本项目：\( e = \mathtt{0x10001} = 65537 \)（费马素数 \( F_4 \)，加密常用），\( n \) 是页面 `publicKeyModulus` 那 256 个十六进制字符（1024 位）。

### 6.3 这套锐捷实现的"非标准"打包（关键）

标准 RSA（如 PKCS#1 v1.5 / OAEP）会给明文加**随机填充**，使同一明文每次密文不同，抵御重放与选择明文攻击。**锐捷 `security.js` 用的是"教科书 RSA"——无填充**，还有几个自定义步骤（脚本 `Invoke-RuijieRSA` 精确复刻）：

1. **拼接并绑定 MAC**：`payload = 明文密码 + ">" + mac`。把 MAC 编进密文，等于把凭据**绑定到当前设备会话**。
2. **整串反转**：`reverse(payload)`（一个简单的混淆步骤）。
3. **分块**：块大小 `chunkSize = 2 × biHighIndex(n)`。1024 位的 \( n \) 有 64 个 16 位"位"，最高位下标 63，故 `chunkSize = 126` 字节。不足整数倍时**补零字节**。
4. **小端打包成大整数**：每块内，`digit[j] = a[2j] + (a[2j+1] << 8)`，整块即该 126 字节的**小端（little-endian）无符号整数**。因 \( 2^{16\cdot63} < n \)，保证块值 \( < n \)。
5. **逐块加密**：\( c = \text{block}^{e} \bmod n \)（`security.js` 用 Barrett 约减，等价于模幂）。
6. **输出**：每块转十六进制，用空格连接。

PowerShell 复刻要点：`[System.Numerics.BigInteger]::new(byte[])` 按小端解释、末尾补一个 `0x00` 保证被当正数；`[BigInteger]::ModPow(block, e, n)` 做模幂；`.ToString("x")` 转十六进制。

### 6.4 为什么能"离线逐字节证明"

正因为**无随机填充 → 加密是确定性函数**（同输入必同输出）。于是我们不必连网试错：让浏览器对同一个 `密码>mac` 加密得到密文 A，脚本算出密文 B，**A、B 逐字节相等即证明复刻 100% 正确**。这是本项目定位问题最有力的手段——把"猜"变成"证"。

> 代价：教科书 RSA 无填充是**密码学弱实现**（确定性、可延展），但这是厂商的设计，我们只能照着复现。

## 7. 字符编码：码点、码元、字节，以及乱码的确切成因

### 7.1 三个层次

- **码点（code point）**：Unicode 给每个字符的编号，如 "密" = U+5BC6。
- **编码（encoding）**：把码点序列化成**字节**的规则。
- **UTF-8**：变长，ASCII 占 1 字节；U+0800–U+FFFF（含绝大多数汉字）占 **3 字节**，位布局 `1110xxxx 10xxxxxx 10xxxxxx`。"密"(U+5BC6) → `E5 AF 86`。
- **GBK**：中文 Windows 传统编码，一个汉字 **2 字节**。

**核心矛盾**：存和读用不同编码表就乱码。计算机底层只有字节，字节本身不带"我是什么编码"的信息。

### 7.2 本项目的两处乱码

1. **日志里的 `å¯ç éè¯¯`**：服务器返回的 JSON 里 `message` 是 UTF-8 的"密码错误"。`Invoke-WebRequest` 在无 / 错 `Content-Type` 时用了错误的单字节编码去解这串 UTF-8 字节，就成了西欧字符乱码。它不影响逻辑（我们判成功只看 ASCII 的 `"result":"success"`），但会误导人——所以要用 `[Encoding]::UTF8` 重新读日志才看到真正的"密码错误"。

2. **`.ps1` 解析报错**：脚本存 UTF-8 **无 BOM**。PowerShell 5.1 打开 `.ps1` 时，若无 BOM，就**回退到系统 ANSI 代码页**（中文 Windows 是 936 = GBK）去解码。中文注释的 UTF-8 三字节被 GBK 拆成一个半汉字 + 残字节，残字节里恰好出现 `"`、`` ` ``、`&` 等，破坏了字符串 / 语法结构 → `ParseFile` 报一堆假错。

### 7.3 BOM 的作用

**BOM（字节顺序标记）** = 文件开头的 `EF BB BF`（UTF-8 编码的 U+FEFF）。它是给读取程序的显式信号："我是 UTF-8"。加上后 PowerShell 5.1 就正确按 UTF-8 解码。

> 规则：**这个 `.ps1` 必须存成 "UTF-8 with BOM"**。而 `.md` / `.json` 等给现代工具读的文件用无 BOM 的 UTF-8 更通用——是否要 BOM 取决于"谁来读"。

## 8. `Invoke-WebRequest` 为何失败、`curl.exe` 为何成功

现象极其反直觉：请求体（body）与浏览器**逐字节相同**、也带了 Cookie 和 `Referer`，服务器却一直判"密码错误"；而把浏览器请求原样交给 `curl.exe` 重放，成功。

根因在传输层。`Invoke-WebRequest`（PowerShell 5.1）底层是 .NET 的 `HttpWebRequest`。.NET 把一部分请求头列为**受限头（restricted headers）**——如 `Referer`、`Host`、`User-Agent`、`Content-Length` 等，**不允许经普通的 header 集合直接设置**，必须走特定属性 / 方法，否则会被忽略或改写。结果我们以为设了 `Referer`，实际发出去的请求头和浏览器不一致，被服务器的校验逻辑拒绝（且只笼统回"密码错误"）。

`curl.exe`（Windows 10 1803+ 自带于 `C:\Windows\System32\`）是"你让带什么头就一字不差带什么头"的工具，行为可预测。所以把登录交给它，原样复刻浏览器的 Cookie 与请求头，一次通过。

**方法论**：当"内容都对却失败"时，抓一份**已知成功的真实请求**（浏览器 F12 → Network → Copy as cURL）作为标准答案，逐字段 / 逐头比对；再用该请求**原样重放**验证可复现性——若重放成功而你的代码失败，问题必在"你的工具如何构造请求"，而非请求内容。

## 9. DPAPI：密码为何能"安全落盘"

明文密码写进脚本 = 谁看到脚本就知道你密码。**DPAPI（数据保护接口）**是 Windows 提供的本地加密服务（底层 `CryptProtectData` / `CryptUnprotectData`）：

- 加密所用的主密钥**从你的 Windows 账户凭据派生**，由系统托管；
- 我们把密码交给它加密，存成 `.campuspwd`；运行时再解密，文件里全程无明文；
- **绑定"同机 + 同用户"**：换机器 / 换 Windows 账户都解不开，别人拷走文件无用。

作用域有两种：`CurrentUser`（默认，绑用户）与 `LocalMachine`（绑机器）。脚本用 `ConvertFrom/To-SecureString`，即 `CurrentUser`。这也解释了计划任务的约束——**必须以存密码时的同一 Windows 账户运行**，否则解密失败。

## 10. 计划任务与登录类型：无人值守的最后一环

**Windows 任务计划程序（Task Scheduler）**是系统级的触发 - 执行框架。我们配了两个触发器：

- `AtLogOn`：用户登录时；
- 每 30 分钟一次的重复触发。

**为什么是轮询而非"联网事件"**：校园网每天掉线是**认证过期**，第二层 WiFi 关联并没断，系统不会发 `NetworkProfile` 的"已连接"事件（事件 ID 10000）。故只能定时探测；脚本开头 `Test-Online` 使"已在线"时立即退出，轮询几乎零开销。

**登录类型（logon type）与 DPAPI 的联动**：

- **Interactive（默认）**：仅当该用户登录时运行（锁屏也算登录）。有完整用户令牌，DPAPI 正常。日常够用。
- **Password**：保存 Windows 账户密码，注销 / 未登录也能运行；DPAPI 正常（有密码可解主密钥）。
- **S4U（Service For User）**：不存密码也能后台运行，但其令牌**无法解开 `CurrentUser` 作用域的 DPAPI 主密钥**（缺密码）——所以我们的 `.campuspwd` 在 S4U 下会解密失败。这是选型时的关键约束。

## 11. 全链路串讲

`campus-net-api.ps1` 单次运行：

```
1. 确认连着校园网 WiFi（netsh wlan）
2. Test-Online：已能上网？→ 是则立即退出
3. GET 一个公网 IP → AC 劫持返回 JS 跳转 → 正则抠出最新 queryString(含 mac/wlanuserip)   [第 3 节]
4. 从 .campuspwd(DPAPI) 解出明文密码                                                       [第 9 节]
5. 按锐捷规则加密：RSA(reverse(密码 + ">" + mac)) 分块 → 十六进制                          [第 6 节]
6. 组装表单：userId / password(密文) / service=shu / queryString(双重编码) / passwordEncrypt=true  [第 4 节]
7. curl.exe：先 GET index.jsp 取 JSESSIONID，再 POST 登录(带全套 Cookie + 请求头)          [第 5、8 节]
8. 解析返回 JSON："result":"success" → 成功；否则按 message 处理、重试
```

| 环节 | 底层原理 | 章节 |
|---|---|---|
| 卡在认证 | 网络分层 + AC 会话表 | 1 |
| 抓 queryString | HTTP 报文 + 劫持重定向 | 2、3 |
| 组装请求 | 百分号编码 / 双重编码 | 4 |
| 加密密码 | RSA 数学 + 非标准打包 | 6 |
| 发送登录 | Cookie / 会话、受限请求头 | 5、8 |
| 存密码 | DPAPI 作用域 | 9 |
| 自动运行 | 计划任务 / 登录类型 | 10 |
| 排乱码 | 编码 / BOM | 7 |

## 12. 调试方法论（最可迁移的部分）

1. **报错信息可能是笼统的**——"密码错误"实际跨了账号、service、queryString、传输层四个根因。不要被字面误导。
2. **取"已知正确"当基准**：浏览器 F12 抓成功请求，逐字段比对，一次暴露"账号多一位""service 空了""queryString 截断"。
3. **能离线证明就不上网试**：确定性函数（如无填充 RSA）用逐字节比对证明，把"猜"变"证"。
4. **分层排除**：先证请求体一致（排除内容）→ 再查传输层（Cookie / 受限头）→ 用 curl 重放证明可复现 → 锁定工具本身。
5. **一次只改一处并立即验证**：避免多改动叠加、无法归因。
6. **环境细节会杀人**：代码页（936/GBK）、BOM、受限头、DPAPI 作用域、登录类型——这些"非算法"因素占了本项目一半的坑。
